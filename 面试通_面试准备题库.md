# 面试通项目面试准备题库

## 一、面试官高频 30 问 + 参考回答

### 1. 先简单介绍一下你的项目
这是一个面向求职场景的面试刷题平台，核心业务包括题库管理、题目管理、关键词检索、在线刷题和刷题记录统计。项目后端基于 Spring Boot，数据层用 MySQL，围绕搜索性能、高并发场景和系统可用性做了专项优化，比如用 Elasticsearch 替代 MySQL 模糊查询、用 Redis BitMap 做签到统计、用 Sentinel 做限流熔断、用 HotKey 和本地缓存应对热点访问，同时还接入了大模型实现 AI 模拟面试。

### 2. 你这个项目最核心的业务链路是什么？
最核心的链路是“用户搜索题目并进入刷题”。因为这个链路访问频率最高、对响应速度最敏感，所以我主要围绕它做了检索优化、缓存优化和限流降级。具体来说，搜索阶段优先走 Elasticsearch，题目详情和题库查询阶段引入 Caffeine 本地缓存和 HotKey 热点探测，异常时通过 Sentinel 和本地缓存兜底。

### 3. 为什么题目搜索不用 MySQL like，而要引入 Elasticsearch？
MySQL 的 like 查询在数据量上来后性能会明显下降，而且中文分词能力较弱，不适合复杂搜索场景。Elasticsearch 的倒排索引更适合关键词检索，结合 IK 分词器后可以支持更自然的中文搜索，在查询性能、召回能力和扩展性上都比 MySQL 模糊查询更适合这个场景。

### 4. 你是怎么做中文分词搜索优化的？
我在 Elasticsearch 里为题目标题、内容等字段绑定了 IK 分词器，让搜索关键词先经过中文分词再进行倒排索引匹配。这样相比单纯把整段文本作为字符串匹配，能够明显提升中文检索的召回效果和用户体验，比如搜索一个技术关键词时，能更准确匹配到相关题目。

### 5. 你提到“动静分离策略”构建题目索引，是什么意思？
我的理解是把变化频率低、适合检索的核心题目信息放到 ES 索引里，比如标题、内容、标签、难度等；而高频变化、强事务相关或者关系型更强的数据，仍然保留在 MySQL 中。这样可以减少 ES 的同步压力，避免每次小变更都触发大范围重建索引，同时也降低了数据不一致带来的维护成本。

### 6. MySQL 和 Elasticsearch 的数据一致性怎么保证？
我采用的是“异步最终一致性”方案。主写库还是 MySQL，Elasticsearch 只负责检索。通过 Spring Scheduler 定时扫描 MySQL 中的变更数据，然后按唯一 ID 同步到 ES。这样可以在保证主业务写入稳定性的同时，兼顾检索效率。虽然不是强一致，但对于题目搜索场景来说，最终一致性是可以接受的。

### 7. 为什么你选择定时同步，而不是每次写库后同步 ES？
如果每次写库都同步 ES，业务链路会更重，写请求的失败面也会扩大，一旦 ES 不可用会直接影响主流程。定时同步把主业务写入和检索同步解耦了，主写链路更稳定，失败重试和补偿也更容易做。这个设计本质上是在性能、一致性和系统复杂度之间做平衡。

### 8. 用唯一 ID 同步 ES 有什么价值？
唯一 ID 可以保证 MySQL 和 ES 中的数据有稳定映射关系。同步时无论是新增、更新还是覆盖写入，都可以直接基于主键进行 upsert，避免因为重复插入导致索引脏数据，也方便后续做问题排查和补偿。

### 9. 如果 Elasticsearch 挂了，你的系统怎么办？
搜索能力会受影响，但我不会让整个系统不可用。对于核心查询接口，我会在 Sentinel 的限流和熔断机制下做降级处理，异常时优先走本地缓存兜底；如果是检索接口，也可以根据场景回退到 MySQL 的基础查询能力，保证用户至少还能访问到主要内容，只是体验和性能会下降。

### 10. 你怎么理解这个项目里的“最终一致性”？
最终一致性的意思是主库 MySQL 和检索库 ES 在某个瞬间可能存在短暂不一致，但系统会通过异步任务在可接受时间内把它们同步到一致状态。这个项目里搜索不是资金类或交易类场景，不要求每次更新后立刻全局可见，所以最终一致性是比较合适的工程选择。

### 11. 为什么刷题记录统计用了 Redis BitMap？
因为刷题记录本质上非常适合表示成“某天是否刷过题”的布尔状态。用户一年最多 365 天，BitMap 可以用极低的空间成本记录每天是否刷题，同时还能快速做按日查询、统计和日历展示。相比一条条存 MySQL 记录或者 Redis 普通字符串集合，BitMap 在空间和查询效率上都更优。

### 12. Redisson 在你的项目里解决了什么问题？
Redisson 主要是让我更方便地使用 Redis 的高级数据结构和分布式能力，比如 BitSet、分布式对象封装等。相比自己手写 Redis 命令，Redisson 提供了更高级、可读性更好的 API，也降低了并发场景下使用 Redis 的复杂度。

### 13. 为什么要引入 Caffeine 本地缓存？
本地缓存的主要价值是减少热点请求直接打到 Redis 或数据库，降低远程调用开销。对于题库详情、热门题目列表这类高频读、变更相对少的数据，本地缓存能显著降低响应时间。在限流或降级场景下，本地缓存还可以作为兜底数据源，增强系统韧性。

### 14. 本地缓存和 Redis 缓存分别适合什么场景？
Redis 更适合做分布式共享缓存，多实例之间数据一致，适合放公共热点数据。Caffeine 更适合单机热点数据加速，访问快、成本低，适合做第一层缓存。我的项目里更偏向用 Redis 做共享能力，用 Caffeine 做热点加速和降级兜底，形成两级缓存思路。

### 15. HotKey 在项目里是怎么发挥作用的？
HotKey 的核心作用是自动识别热点 key，把真正高频的数据提前提升到本地缓存层。这样在瞬时高并发场景下，请求不需要每次都打到 Redis 或数据库，可以优先命中本地缓存，从而防止热点数据把后端资源打穿。

### 16. HotKey 和普通缓存预热有什么区别？
普通缓存预热通常依赖人工经验或定时任务，适合已知热点；HotKey 更偏动态识别，它能根据真实访问行为发现当前热点并自动处理，所以更适合访问模式不固定、热点变化快的业务场景。两者相比，HotKey 的动态性更强，但也需要配套治理和监控。

### 17. 你怎么防止数据库被瞬时流量击穿？
我主要从三层做防护。第一层是热点识别和缓存命中，用 Caffeine 和 HotKey 尽可能把请求挡在应用层；第二层是 Redis 做共享缓存，避免每个请求都落到数据库；第三层是 Sentinel 做限流和熔断，保证在异常流量或依赖故障时系统能主动降级，而不是把数据库拖死。

### 18. 你项目里 Sentinel 主要用在什么地方？
主要用于核心查询接口，比如题目列表、题库列表这类高频访问接口。我结合 Sentinel 注解和 Dashboard 做规则管理，对高频请求进行限流、熔断和降级，在接口异常率上升或者响应时间过长时主动保护服务。

### 19. 限流、熔断、降级分别是什么意思？
限流是控制单位时间内允许进入系统的请求量，防止流量过大。熔断是当接口异常率或慢调用比例超过阈值时，暂时切断部分请求，避免故障扩散。降级是在系统压力过大或依赖异常时，主动返回简化结果、缓存数据或兜底结果，优先保证服务可用。

### 20. 你在项目里是怎么做降级兜底的？
我的做法是把热点查询结果缓存在本地，正常情况下走实时查询，异常情况下通过 Sentinel 触发降级，直接返回本地缓存中的可用数据。这样虽然可能牺牲一点实时性，但能保证核心功能在高压和异常场景下依然可访问。

### 21. Sentinel Dashboard 在你项目里的价值是什么？
Dashboard 的价值主要有两个。第一是可视化查看流量、规则和触发情况，方便运维和排查；第二是支持动态调整规则，不需要每次改代码再上线。这样在实际压测或线上波动时，可以快速调节限流阈值和降级策略。

### 22. 你怎么做访问频控和自动封禁？
我基于 Redis 做访问计数，使用 Lua 脚本保证计数和过期设置的原子性。对于短时间内高频访问的用户或 IP，会先触发告警，再根据阈值进行封禁处理。这样既能应对简单的恶意刷接口行为，也能避免非原子操作导致计数不准。

### 23. 为什么频控要用 Lua 脚本，而不是普通的 Redis incr？
如果只用普通 incr，再分开设置过期时间，会有并发下的竞态问题，比如 key 已经递增但没及时设置过期。Lua 脚本可以把“计数 + 过期 + 判断”放到 Redis 端一次完成，保证原子性，减少并发下的边界问题。

### 24. 这个项目里最能体现高并发设计意识的地方是什么？
我认为主要有三点。第一是把搜索从 MySQL 转向 ES，降低关系库检索压力。第二是刷题记录统计采用 Redis BitMap，兼顾性能和空间效率。第三是通过 Caffeine、HotKey、Sentinel、Redis 访问频控组成分层防护体系，让系统在热点流量和异常流量下仍然可控。

### 25. Spring Scheduler 在项目里具体做了什么？
主要负责 MySQL 到 Elasticsearch 的定时同步。它会周期性扫描变更数据并同步到 ES，保证搜索索引能逐步追上主库状态。这个任务的设计重点是幂等和稳定性，所以我会基于唯一 ID 来更新索引，避免重复写入。

### 26. 如果定时同步失败了，你怎么处理？
首先任务本身要具备幂等性，这样失败后可以安全重试。其次需要记录同步日志和异常信息，便于排查具体失败原因。再往后如果做得更完整，可以加失败重试、死信记录或人工补偿入口。这个项目里我重点体现的是“主写链路不受影响”和“同步任务可重跑”。

### 27. AI 模拟面试这个功能的核心设计是什么？
核心设计是“题库驱动 + Function Calling + 结构化输出”。大模型并不是完全自由发挥，而是先基于题库和难度约束选题，再结合标准答案进行点评，最后输出结构化面试报告。这样可以兼顾真实性、可控性和结果可复用性，避免模型输出过于发散。

### 28. 为什么 AI 模拟面试要用 Function Calling？
因为 Function Calling 更适合把模型能力和业务系统结合起来。模型负责理解上下文和组织表达，系统函数负责选题、拿标准答案、生成报告数据。这样可以减少纯 Prompt 方案的不稳定性，让输出更结构化，也更容易和项目中的题库、难度、答案等业务数据结合。

### 29. AI 面试报告为什么要做成结构化？
结构化的价值在于可展示、可存储、可复盘。比如我可以把报告拆成问题表现、知识点掌握情况、薄弱点、综合建议等字段，后续可以支持用户查看历史记录、对比成长趋势，也更方便前端直接渲染，而不是只能展示一段纯文本。

### 30. 如果让你再优化一次这个项目，你会优先做什么？
我会优先做三件事。第一，进一步完善 ES 数据同步链路，比如增加失败补偿和更细粒度的同步标记。第二，完善缓存一致性策略，明确哪些数据走本地缓存、哪些数据走 Redis，并补充监控。第三，把 AI 面试模块的可观测性做强，比如增加耗时、成功率和错误类型统计，让系统更适合真实线上场景。

## 二、项目亮点的 3 分钟自我介绍稿

面试官您好，我想介绍一下我做的一个项目，叫“面试通”，它是一个面向求职场景的面试刷题平台，主要解决的是题目管理、题库管理、关键词检索、在线刷题和刷题记录统计这些典型业务问题。

这个项目的后端主要基于 Spring Boot，数据存储用了 MySQL 和 Redis，同时为了优化检索能力，我引入了 Elasticsearch。因为刷题平台最核心的高频场景其实不是写入，而是“搜索题目”和“高频查询题库、题目详情”，所以我的设计重点主要放在三个方向上。

第一个方向是搜索和检索优化。我没有直接使用 MySQL 的 like 模糊查询，而是搭建了 Elasticsearch 检索体系，并且给题目相关字段绑定了 IK 分词器，来提升中文搜索的效果和性能。另一方面，我没有把所有业务数据都硬塞进 ES，而是采用了动静分离策略，把适合检索、相对稳定的数据放进索引，把高频变更和关系型更强的数据仍然保留在 MySQL，这样能降低索引同步成本。

第二个方向是数据同步和高并发性能优化。因为主写库还是 MySQL，所以我通过 Spring Scheduler 定时把 MySQL 的变更数据同步到 ES，并且基于唯一 ID 做索引更新，保证同步过程具备幂等性和可追踪性。针对高并发场景，比如刷题记录统计，我使用了 Redis BitMap 和 Redisson 来实现用户年度刷题记录的高效存储和查询；针对热点题库和热点查询，我又引入了 Caffeine 本地缓存和 HotKey 热点探测机制，尽量把高频请求挡在缓存层，避免数据库在瞬时流量下被击穿。

第三个方向是系统可用性保障。我对核心查询接口接入了 Sentinel 和 Dashboard，做限流、熔断和降级。当接口异常率高或者压力过大时，会优先通过本地缓存返回兜底数据，保证核心功能不完全不可用。另外，在安全性上，我还结合 Redis 访问频控、Lua 脚本原子计数和自动告警封禁机制，处理恶意高频访问的问题。

除了这些传统后端优化外，我还做了一个 AI 模拟面试模块。它基于大模型 Function Calling 实现，支持按题库和难度动态选题，结合标准答案做针对性点评，并自动生成结构化面试报告。这个模块我比较强调“可控性”，也就是让大模型不是完全自由发挥，而是跟题库、答案和报告模板结合起来，真正服务于业务。

整体来说，这个项目让我比较有收获的地方，不只是把功能做出来，而是围绕搜索性能、高并发设计、可用性保障和 AI 与业务结合做了一整套工程化思考。

## 三、针对八股追问的速记版答案

### 1. 为什么 ES 比 MySQL like 更适合搜索？
因为 ES 基于倒排索引，检索复杂文本时性能更稳定，适合大数据量下的关键词搜索；MySQL like 本质上更偏字符串匹配，数据量大后成本高，而且中文分词能力弱。

### 2. 什么是倒排索引？
倒排索引可以理解成“词到文档”的映射。传统索引更像“文档到词”，而倒排索引能让搜索系统快速知道某个词出现在哪些文档里，所以很适合全文检索。

### 3. IK 分词器的作用是什么？
IK 分词器是面向中文文本的分词工具，能把一句中文拆成更适合检索的词项，提高搜索召回效果和匹配准确度。

### 4. 为什么 MySQL 和 ES 通常采用最终一致性？
因为 ES 更多是检索服务，不适合作为强事务主存储。让 MySQL 做主库、ES 做检索库，可以降低主流程复杂度，换来更好的性能和可扩展性。

### 5. 定时同步的优点是什么？
优点是解耦主写链路、降低业务请求延迟、方便失败重试和补偿。缺点是数据会有短暂延迟，不是强一致。

### 6. BitMap 适合什么场景？
适合大量布尔状态的记录，比如签到、是否在线、是否完成某任务。优点是极省空间、查询快。

### 7. Redisson 的价值是什么？
Redisson 对 Redis 做了更高层封装，使用分布式对象、锁和高级数据结构更方便，代码可读性和开发效率更高。

### 8. Caffeine 和 Redis 的区别是什么？
Caffeine 是本地缓存，速度快但不共享；Redis 是分布式缓存，适合多实例共享数据。前者更适合热点加速，后者更适合公共缓存。

### 9. HotKey 解决什么问题？
解决热点 key 被高频访问时，缓存层和数据库被打穿的问题。它通过识别热点并下沉到本地缓存，减少远程依赖压力。

### 10. 什么是缓存击穿？
某个热点 key 失效后，大量请求同时落到数据库，导致数据库瞬时压力暴涨。

### 11. 什么是缓存穿透？
请求访问的数据本来就不存在，导致缓存不命中且持续打到数据库。

### 12. 什么是缓存雪崩？
大量缓存同一时间失效，导致请求集中落到后端存储，引发整体雪崩。

### 13. 限流的核心目的是什么？
控制流量进入速度，防止系统被突发请求压垮。

### 14. 熔断的核心目的是什么？
在依赖异常或接口异常率过高时，及时切断故障链路，防止问题扩散。

### 15. 降级的核心目的是什么？
在资源紧张或依赖异常时，优先保证服务可用，哪怕牺牲部分功能、实时性或精度。

### 16. Sentinel 和 Hystrix 的思路差别怎么说？
都能做熔断降级，但 Sentinel 更强调流量治理、规则丰富和可视化控制，在国内 Java 生态里用得比较多。

### 17. Lua 脚本为什么适合做 Redis 原子操作？
因为 Lua 脚本在 Redis 服务端一次执行完成，可以把多个操作合成一个原子过程，避免并发下的数据竞争。

### 18. Spring Scheduler 适合做什么？
适合周期性任务，比如同步数据、定时清理、统计聚合等。优点是接入简单，适合中小型项目。

### 19. 幂等性为什么重要？
因为异步重试、定时任务和分布式场景下，操作很可能重复触发。幂等性可以保证多次执行的结果和一次执行一致。

### 20. Function Calling 的价值是什么？
它让大模型不只是生成文本，而是能够按业务规则调用后端函数，从而实现“模型理解 + 系统执行”的结合，输出更可控。

### 21. 为什么 AI 场景要强调结构化输出？
结构化输出更方便展示、存储、分析和后续复用，也更适合和业务系统对接。

### 22. 为什么你要把题库、难度这些条件交给系统控制，而不是完全让模型自由发挥？
因为业务系统需要可控性和一致性。完全靠模型自由发挥容易发散，而系统控制选题范围、难度和答案来源后，结果更稳定。

### 23. 什么是系统可用性？
就是系统在正常和异常情况下，都尽可能持续提供可接受服务的能力。

### 24. 你的项目里可用性是怎么体现的？
通过限流、熔断、缓存兜底、热点保护和访问频控，做到“有问题时先保护系统，再尽量保核心功能”。

### 25. 如果面试官问你这个项目最大的技术亮点是什么，怎么答？
我会回答是“围绕高频搜索链路做了完整的性能和可用性设计”，不是单点用了某个框架，而是把 ES、缓存、HotKey、Sentinel、Redis 频控和 AI 模块串成了完整业务闭环。

## 四、最后的使用建议

### 1. 面试时最建议优先展开的三个点
第一讲 ES 搜索优化，因为它最能体现你对业务瓶颈的判断。第二讲高并发缓存设计，因为它最能体现工程意识。第三讲 Sentinel 和频控，因为它最能体现你对可用性的理解。

### 2. 回答时的推荐结构
建议按“业务场景、原始问题、设计方案、方案权衡、最终收益”这个结构来讲，这样比单纯背八股更像真实做过项目。

### 3. 哪些问题不要答太虚
不要只说“提高性能”“增强可用性”“优化体验”，一定要落到具体技术动作，比如为什么换 ES、为什么用 BitMap、为什么做本地缓存兜底、为什么用 Lua 保证原子性。
