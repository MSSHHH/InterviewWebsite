## 一、项目整体与职责

### 1. 你先用 1 分钟介绍一下这个项目，核心业务和你的职责是什么？

这个项目叫“面试通”，本质上是一个面向求职场景的面试刷题平台。它的核心业务分成两端：一端是管理员侧，主要负责题库、题目和后台管理，比如题目录入、批量管理、题库维护；另一端是用户侧，主要是按关键词搜索题目、在线刷题、查看刷题记录日历，以及基于题库做 AI 模拟面试。

从技术设计上，我没有把它只当成一个普通 CRUD 项目来做，而是围绕三个重点做了专项设计。

第一是搜索检索优化。题目搜索是高频核心链路，所以我用 Elasticsearch 替代 MySQL 模糊查询，并接入 IK 分词器提升中文检索效果。

第二是高并发性能优化。比如用户年度刷题记录统计我使用了 Redis BitMap，热点查询场景我结合了 Redis、Caffeine 和 HotKey 做多层缓存，避免热点流量直接打穿数据库。

第三是系统可用性保障。我对核心接口接入了 Sentinel 做限流、熔断和降级，同时结合 Redis 访问频控、Lua 原子计数和自动告警封禁来处理恶意访问和异常流量。

如果说我的职责，我主要负责的是后端核心链路设计和性能优化，包括搜索架构、MySQL 到 ES 的同步方案、缓存和热点治理、限流降级，以及 AI 模拟面试模块里 Function Calling 与题库数据的结合。

### 2. 这个项目的主要用户角色有哪些？题库、题目、题解、刷题记录这几个核心对象之间是什么关系？

这个项目主要有两类角色。

第一类是管理员。管理员负责维护平台内容，包括创建题库、管理题目、批量把题目加入题库、管理后台资源等。管理员是内容供给方。

第二类是普通用户。普通用户的主要行为是搜索题目、进入题库刷题、查看题目详情、做刷题记录统计，以及参与 AI 模拟面试。

几个核心对象之间的关系可以这样理解：

1. 题库和题目是多对多关系。
一个题库里可以有很多题目，一个题目也可以被收录到多个题库，所以我没有直接在题目表里放 `questionBankId`，而是单独设计了题库题目关联表。

2. 题目和题解通常是一对一或者一对多的内容关系。
从业务上讲，题目是核心主体，题解是围绕题目的解析内容。即使题解单独存在，它的从属关系也很明确，是服务于题目理解和复盘的。

3. 用户和刷题记录是一对多关系。
一个用户会产生很多刷题记录，刷题记录本质上反映的是“用户在某一天是否刷过题，或者刷了哪些题”。如果做年度日历统计，这类数据特别适合压缩成 BitMap。

4. 用户和题库之间还会通过刷题行为产生间接关系。
比如用户在某个题库中刷题、完成模拟面试、查看历史报告等，其实都是围绕“用户 - 题库 - 题目”这个三角关系展开的。

所以这个项目虽然看上去是内容平台，但本质上是一个“题库组织内容、题目承载知识点、用户行为沉淀学习轨迹”的业务系统。

### 3. 为什么这个项目要做成前后端分离？为什么前端选 Next.js，而不是普通 React SPA？

我选择前后端分离，核心原因有三个。

第一，职责边界更清晰。后端可以专注于业务逻辑、检索、缓存、可用性和接口治理，前端可以专注于页面渲染、交互体验和状态管理。这样后续无论是独立部署还是多人协作，都会更顺畅。

第二，更适合内容平台这类产品形态。刷题平台有很多页面，比如题库页、题目详情页、搜索结果页，这些页面既要求交互体验，也希望首屏加载尽量快，所以分离后更容易做专项优化。

第三，更利于接口复用。后端提供统一 API 后，不光 Web 前端能用，后面如果要做小程序、App 或者 AI 侧工具调用，也有统一的数据出口。

前端为什么选 Next.js 而不是普通 React SPA，我主要从两点考虑。

第一，SSR 对内容型平台更友好。像首页、题库列表、题目详情这类页面，SSR 能提升首屏渲染体验，也更利于 SEO。如果全做成纯 SPA，首次进入时更依赖前端 JS 拉取数据，白屏时间和首屏体验通常会更差。

第二，Next.js 的工程组织能力更强。它本身对路由、服务端渲染、静态生成和混合渲染的支持比较完整，比自己在 React SPA 里再额外拼这些能力更顺手。对一个内容平台来说，Next.js 比单纯的 React SPA 更适合作为中大型前端容器。

如果一句话概括，我会说：前后端分离解决的是协作和职责问题，Next.js 解决的是内容平台的渲染和工程化问题。

### 4. 你这个项目里哪些功能是最核心的高频链路？哪些接口最容易成为性能瓶颈？

我认为最核心的高频链路有两条。

第一条是“搜索题目 -> 查看题目列表 -> 进入题目详情”。这是用户最常用、访问频率最高的业务路径，也是最直接影响体验的一条链路。

第二条是“进入题库 -> 浏览题目 -> 刷题 -> 查看刷题记录”。这条链路的特点是读多写少，但用户访问比较集中，特别是在高峰期会形成明显热点。

从接口角度看，最容易成为性能瓶颈的主要有三类：

1. 题目搜索接口。
如果用 MySQL like 直接查标题和内容，数据量上来以后性能会迅速下降，而且中文分词效果也不好，所以这是我最早重点优化的接口。

2. 题库详情和题目分页接口。
这类接口访问量大、数据结构相对固定，很容易形成热点。如果没有缓存层，数据库压力会很大。

3. 刷题记录统计接口。
比如按年度展示刷题日历时，如果每次都现算、现查明细记录，随着数据累积会变得低效，所以我用 Redis BitMap 进行压缩和快速统计。

所以我的整体思路不是等接口慢了再优化，而是先识别高频链路，再把搜索、缓存、热点和降级能力优先铺在这些链路上。

### 5. 如果让你选一个最有技术亮点的点展开，你会讲哪个？为什么？

如果让我选一个最有代表性的技术亮点，我会优先讲“搜索链路从 MySQL like 升级为 Elasticsearch 检索体系，并配套做 MySQL 到 ES 的异步同步和降级兜底”。

原因是这个点最能体现我不是在堆技术，而是在围绕业务瓶颈做系统设计。

这个项目里最核心的场景就是题目搜索。刚开始如果直接用 MySQL like，也能做出来，但随着题目量增长，会遇到两个问题：一是查询性能下降，二是中文检索体验差。于是我把检索链路拆开，MySQL 继续做主存储，ES 专门负责搜索；再通过 IK 分词器优化中文搜索效果；同时为了不让 ES 同步成本过高，我又采用动静分离策略，只把适合检索的核心字段放进索引。

接着就会出现第二层问题，也就是一致性和可用性。为了解决这个问题，我引入了定时增量同步，让 MySQL 和 ES 形成最终一致；同时在 ES 出问题的时候，又可以通过 Sentinel 和本地缓存做降级兜底。

所以这个技术点不是一个单点，而是一整套链路设计，能体现我对性能、数据一致性和系统可用性的综合权衡。

### 6. 你做这个项目时，遇到过最难的问题是什么？最后怎么定位和解决的？

我觉得最难的问题其实不是某个 bug，而是“怎么在搜索性能、数据一致性和系统复杂度之间做平衡”。

一开始直觉上最简单的做法，是所有搜索都走 MySQL；但很快我就意识到这会在性能和中文搜索效果上成为瓶颈。后来换成 ES 以后，性能问题解决了一部分，但新的问题又出现了：MySQL 和 ES 的数据怎么保持一致？如果每次写库都同步 ES，会把主业务链路拉长，而且 ES 不可用时还会反向影响主写流程。

我定位这个问题时，先把它拆成了三个子问题：

1. 主写链路要不要依赖 ES？
我的结论是不能强依赖，否则系统耦合度太高。

2. 搜索是不是必须强一致？
在刷题平台这个场景下，不要求像交易系统那样强一致，所以最终一致性是可接受的。

3. ES 出问题时系统要不要完全不可用？
我的结论也是不能，所以要有降级方案。

基于这个分析，我最后采用了“主写 MySQL、异步同步 ES、搜索链路优先走 ES、异常时降级兜底”的方案。技术上就是用 Spring Scheduler 做定时同步，用唯一 ID 保证索引更新准确性，用 Sentinel 和本地缓存处理异常流量和依赖故障。

我觉得这个问题最难的点，不是技术实现本身，而是要先把架构边界和业务容忍度想清楚。

### 7. 如果线上用户量突然涨 10 倍，你最担心哪个模块先出问题？

如果用户量突然涨 10 倍，我最担心的第一个模块其实不是写库，而是高频读链路，也就是搜索接口、题库详情接口和热点题目查询接口。

原因很简单，这个项目的业务形态是明显的读多写少。真正会先被放大的，通常是搜索和浏览请求，而不是管理员写入。尤其是搜索接口，如果直接打 ES，ES 集群压力会上升；如果缓存层设计不合理，热点数据还可能绕过缓存打到 Redis 或 MySQL，形成级联压力。

第二个我会担心的是缓存与热点治理层。如果突然出现爆款题库或热点题目，单个 key 的访问量会非常集中，这时候 Redis 可能先被打热。如果没有 HotKey 和本地缓存，后面数据库和 ES 也会被间接拖垮。

第三个才是数据库本身。数据库一般不是第一时间出问题，而是在缓存命中下降、热点失控、限流失效之后，被高频请求打穿。

所以如果用户量突然放大 10 倍，我的优先观察顺序会是：接口 RT 和 QPS、缓存命中率、Redis 热点分布、Sentinel 触发情况、再到 MySQL 和 ES 的压力指标。

## 二、后端与业务设计常问

### 8. 题目搜索为什么要引入 Elasticsearch？直接 MySQL like 不行吗？

如果只是小数据量、功能验证阶段，MySQL like 当然能做，甚至实现最快。但从业务长期演进来看，它不适合作为正式的搜索方案。

原因主要有三个。

第一，性能问题。MySQL like 对大文本字段做模糊查询时，随着数据量增长性能会明显变差，尤其当搜索字段比较多、过滤条件复杂时，数据库压力会迅速上升。

第二，搜索体验问题。题目搜索本质上是全文检索，而不是精确过滤。MySQL 更擅长结构化查询，不擅长做中文语义层面的分词匹配。比如用户搜“Redis 热 key”，如果只靠普通 like，很难在召回效果上做到理想状态。

第三，可扩展性问题。后续如果要支持相关性排序、分词检索、标签组合搜索、多字段加权这些能力，ES 的能力明显更强。

所以我引入 ES 的本质原因，不是为了“技术栈更高级”，而是因为搜索这个场景天然更适合倒排索引模型，而不是关系型数据库的字符串匹配模型。

---

### 9. 你们 ES 和 MySQL 的数据一致性怎么保证？为什么采用“定时增量同步”？

我的方案是：MySQL 做主存储，ES 做检索副本，通过定时增量同步实现最终一致性。

为什么这么做？因为我要先保证主业务写入稳定。题目新增、修改、上下线这些操作，本质上属于后台管理流程，第一优先级是写库成功，而不是同步 ES 必须立刻成功。如果把 ES 同步放进主写链路，一旦 ES 抖动或超时，就会把主流程一并拖慢，甚至影响后台操作可用性。

采用定时增量同步后，整体链路就变成：

1. 业务数据先写 MySQL；
2. 定时任务扫描变更数据；
3. 根据唯一 ID 对 ES 做新增或覆盖更新；
4. 最终让检索库逐步追平主库。

这里我强调的是“增量”和“唯一 ID”两个关键词。

增量同步可以降低全量同步的成本，避免每次都重刷索引；唯一 ID 则保证同步过程具备幂等性和准确映射关系，即使任务重试也不会产生脏数据或重复文档。

这个方案的代价是会有短暂延迟，但对于面试刷题平台这种搜索业务来说，最终一致性是完全可以接受的。

### 10. 如果 ES 挂了，搜索功能怎么办？你的降级方案是什么？

我不会让 ES 故障直接变成整个系统不可用，而是分层处理。

第一层是 Sentinel 熔断和降级。对于核心高频接口，我会通过 Sentinel 做保护，一旦检测到依赖异常、错误率升高或者慢调用变多，就先限流和降级，防止请求继续堆积。

第二层是本地缓存兜底。对于热点题库、热门题目列表这类可以容忍短暂不那么实时的数据，我会优先返回 Caffeine 本地缓存中的旧值，让用户还能看到主要内容，而不是直接报错。

第三层是按业务选择是否回退到 MySQL 基础查询。严格来说，MySQL 不适合承接高并发全文检索，但在故障恢复期，如果只是提供一个弱化版搜索，比如标题关键字模糊匹配、或者限制分页深度，它可以作为应急兜底方案。

所以我的降级思想不是“ES 挂了什么都不做”，而是“先保护系统，再保证核心功能最小可用，最后再逐步恢复完整搜索能力”。

### 11. 题库和题目的关系为什么单独做关联表？如果一个题属于多个题库怎么处理？

题库和题目是一个典型的多对多关系，所以从建模上就不应该把它硬做成一对多。

一个题库里可以有很多题目，这个很好理解；但反过来，一个题目也可能被多个题库复用，比如“Redis 高频面试题”既可以放在“后端题库”里，也可以放在“中间件题库”里。如果直接在题目表上存一个 `questionBankId`，就会把模型限制死，后面一旦要支持多题库复用，就要大改表结构。

所以我单独设计了题库题目关联表。这样做有几个好处：

1. 能自然表示多对多关系；
2. 方便做批量加题、批量移除题目；
3. 后续如果关联关系上要加扩展字段，比如排序、加入时间、加入人等，也有承载位置；
4. 对 ES 也更友好，因为题目和题库关系如果是高频变更的，我可以保留在 MySQL，不强塞到 ES 文档里。

如果一个题属于多个题库，我就是在关联表中插入多条关系记录，而不是复制多份题目数据。这样既避免数据冗余，也更利于统一维护题目本身。

### 12. 批量给题库加题目时，你为什么做分批、异步、事务处理？这样做的收益和风险是什么？

这个设计背后的核心考虑是：批量操作很容易既拖慢请求，又把数据库事务拉得太长。

如果一次性把大量题目全塞进一个大事务里，会有几个问题：

1. 单次事务太大，数据库压力高；
2. 锁持有时间变长，影响并发；
3. 一旦中途失败，回滚成本很高；
4. 请求线程长时间阻塞，接口 RT 会很差。

所以我会把大批量任务拆成多个小批次，每批独立处理。这样每次事务更短、失败范围更小、数据库压力也更可控。

异步处理的意义在于，不让主请求线程一直卡在那里等所有批次完成，尤其在批量任务较大时，异步化可以提高吞吐和用户感知体验。

事务处理的意义在于保证每个批次内部的一致性。也就是说，我不是追求“全局一个超大事务”，而是追求“每个小批次内部要么成功，要么回滚”，这样在性能和一致性之间更平衡。

当然这个方案也有风险。

第一，异步批处理后，全局不再是严格意义上的原子事务，可能出现一部分批次成功、一部分失败。

第二，线程池和批次大小如果设置不合理，可能反而把数据库打得更猛。

第三，异常收集和补偿要更认真处理，否则问题会更隐蔽。

所以我会把这个方案定义为“用分批事务和异步并行去换吞吐和可用性，但必须接受全局原子性下降，并配套补偿能力”。

### 13. 为什么分页查询要做 VO 封装，而不是直接把数据库实体返回前端？

我一般不会直接把数据库实体暴露给前端，主要原因有三类。

第一是安全性和边界问题。数据库实体通常包含的是存储层字段，而前端真正需要的是展示层字段。直接返回实体容易把一些不该暴露的信息带出去，比如内部状态字段、逻辑删除字段、审计字段等。

第二是接口稳定性问题。数据库表结构是面向存储设计的，前端接口是面向展示和交互设计的。如果把两者绑死，后续只要表结构变化，前端接口就容易被牵连；VO 封装相当于在中间加了一层抗震层。

第三是便于做扩展字段聚合。比如题目分页列表返回时，前端可能还想拿到用户信息、题库信息、标签数组、是否已刷等字段，这些字段不一定天然就在一张表里。通过 VO 封装，我可以把多个来源的数据组装成前端真正需要的结构。

所以 VO 不是为了“多写一层代码”，而是为了把数据库模型和接口模型解耦，让接口更安全、更稳定、更贴近业务。

---

### 14. 你项目里哪些地方用了 AOP / 全局异常处理 / 统一响应封装？这样做的价值是什么？

这三个我一般会一起设计，因为它们本质上都是在做“通用能力下沉”。

第一，AOP。
AOP 适合处理那些和具体业务无关、但又需要统一织入的逻辑，比如权限校验、日志记录、接口耗时统计等。这样可以避免在每个 Controller 里重复写样板代码，也能让核心业务逻辑更干净。

第二，全局异常处理。
像参数错误、权限错误、业务异常、系统异常这类问题，如果让每个接口自己 try-catch，不但代码冗余，还容易风格不一致。通过统一异常处理器，我可以把异常转换成标准错误码和标准响应结构，让前后端更容易协作。

第三，统一响应封装。
统一响应结构的价值是让所有接口都遵循一致的返回格式，比如 `code`、`message`、`data` 这类字段固定。这样前端处理更稳定，接口测试和网关治理也更方便。

如果让我总结，这三者的价值在于：把重复、横切、规范化的能力抽出来，减少散落在业务代码里的样板逻辑，让系统更可维护。

---

## 三、Redis / 缓存 / 高并发高概率追问

### 15. 你为什么同时用了 Redis 和 Caffeine，两级缓存分别解决什么问题？

我同时使用 Redis 和 Caffeine，本质上是在解决两个不同层次的问题。

Redis 解决的是“分布式共享缓存”问题。它适合放多个应用实例都要访问的公共数据，比如题库详情、热门题目列表、刷题记录状态等。它的优点是跨实例共享、容量更大、可统一治理。

Caffeine 解决的是“单机热点加速”问题。它直接存在应用内存里，没有网络开销，访问速度非常快，特别适合承接那些高频且可容忍短暂不一致的热点数据。

两级缓存结合起来以后，思路就变成：

1. 第一层先查本地缓存，尽可能用最低成本命中热点；
2. 本地没有再查 Redis；
3. Redis 也没有再回源数据库；
4. 回源结果再逐层回填。

这样做的核心价值，是把最贵的数据库请求挡在最后面，把最便宜的内存访问放在最前面，形成明显的性能梯度。

---

### 16. 本地缓存和分布式缓存怎么保证一致性？如果数据更新了，本地缓存怎么失效？

严格来说，本地缓存和分布式缓存不可能做到绝对实时一致，所以我的处理思路是“按业务价值做一致性分层”。

对于题库详情、热门题目列表这类数据，我会接受短时间内的最终一致，而不是追求强一致。因为这些数据就算延迟几秒或几十秒，对用户体验影响通常不大，但换来的是更高的吞吐能力。

本地缓存失效通常有几种方式：

1. 主动失效。
比如后台管理员修改了题库或题目后，主动删除对应 key，让下一次访问重新回源加载。

2. TTL 过期。
即使没有主动删除，我也会给本地缓存设置较短的过期时间，避免脏数据长期驻留。

3. 热点探测刷新。
如果结合 HotKey，一些热点数据会随着访问过程被重新识别和刷新。

如果面试官继续追问，我会强调一点：本地缓存不是为了替代 Redis，而是为了缓解热点压力，所以它更适合承接“允许短暂不一致”的读场景，而不适合强事务场景。

---

### 17. 什么场景适合本地缓存，什么场景更适合 Redis？

本地缓存适合三类场景。

第一，高频读取、变更不频繁的数据。比如热门题库、排行榜、热门题目列表。

第二，对响应时间特别敏感的数据。因为本地缓存没有网络开销，命中后几乎就是内存访问。

第三，降级兜底数据。比如某些接口在依赖故障时，可以先返回本地缓存中的旧值。

Redis 更适合四类场景。

第一，多实例共享的数据。比如用户登录态、跨实例都要读取的缓存。

第二，需要借助 Redis 数据结构的场景。比如 BitMap、Set、Hash、Lua 计数等。

第三，需要做分布式协调的场景。比如基于 Redisson 的分布式对象或锁。

第四，需要更大容量或更强可运维性的缓存场景。

所以简单理解就是：本地缓存偏“快”，Redis 偏“共享和能力”。它们不是替代关系，而是互补关系。

---

### 18. 你项目里的签到记录为什么用 BitMap？和普通一条条存记录比，优势是什么？

刷题记录或者签到记录这类数据，有一个很明显的特点：对于很多业务展示来说，真正关心的是“某一天有没有发生”，而不是这一天到底存了多少冗余字段。

以年度刷题记录为例，一个用户一年最多 365 天，我完全可以用 365 个 bit 来表示这一年里每天有没有刷题。这样做的优势非常明显。

第一，空间占用极低。365 个布尔状态压成位图，远比存 365 条明细记录更省。

第二，读取效率高。前端做日历展示时，我可以快速按位读取某个月或者某一年的刷题情况。

第三，统计方便。比如统计某个月刷题天数、全年连续活跃等，都可以在位图基础上做进一步运算。

如果改成普通一条条存记录，虽然可读性更直观，但会带来存储冗余、查询聚合成本高、统计效率差等问题。所以在这个业务里，BitMap 是非常贴合场景的。

---

### 19. BloomFilter 为什么适合做黑名单/防刷场景？它的缺点是什么？

BloomFilter 适合黑名单和防刷场景，核心原因是它非常省空间，而且查询一个元素“可能存在还是一定不存在”的速度非常快。

像 IP 黑名单、恶意用户标记、疑似爬虫标记这类集合，通常特点是数量可能比较大，但每次判断逻辑又很简单，只需要快速判断“要不要拦”。这时候 BloomFilter 非常合适，因为它能用很低的内存成本支撑快速判定。

但它也有明确缺点。

第一，它有误判率。也就是本来不在集合里的元素，可能被判断为在集合里。

第二，它不适合频繁删除。传统 BloomFilter 对删除支持不好，删除容易影响其他位。

第三，它通常更适合作为第一层快速过滤，而不是最终裁决依据。

所以如果我把它用在黑名单和防刷场景里，我的定位一般是“快速粗筛”，而不是“绝对精确的最终事实来源”。真正涉及强业务判断时，还需要配合更精确的数据源或更高层规则一起使用。

---

### 20. 你在项目里哪里用了 Redisson？为什么不用自己写 Redis 分布式锁？

我在项目里主要用 Redisson 去操作 Redis 的高级数据结构能力，比如 BitSet 这种更贴近业务语义的封装。它的好处是 Java API 比较自然，可读性强，而且减少了直接操作底层 Redis 命令时的一些样板代码。

如果扩展到分布式锁场景，我也更倾向于用 Redisson，而不是自己从零手写。

原因主要有三个：

第一，自己写分布式锁容易踩边界问题，比如锁续期、超时释放、可重入性、异常场景下的锁清理等，这些都不是简单 `setnx + expire` 就能彻底解决的。

第二，Redisson 已经把很多复杂细节做了封装，稳定性和成熟度更高。

第三，项目里真正重要的是业务正确性，而不是为了“展示会手写分布式锁”去重复造轮子。

所以我的原则一般是：如果业务场景已经适合使用成熟组件，就优先选择经过验证的方案，而不是为了炫技增加隐患。

---

### 21. 什么是缓存穿透、缓存击穿、缓存雪崩？你这个项目分别怎么防？

这三个概念很容易混，但在面试里一定要讲清楚。

缓存穿透，指的是请求访问的数据本来就不存在，导致每次都查不到缓存，也查不到数据库，请求持续打到数据库。

缓存击穿，指的是某个热点 key 在失效瞬间，大量请求同时回源数据库，把数据库打爆。

缓存雪崩，指的是大量缓存同一时间失效，或者缓存服务本身异常，导致大量请求一起压向后端存储。

我在项目里的防法分别是：

1. 防穿透。
对非法请求做参数校验，对明显不存在的对象可以做空值缓存；在黑名单或风险过滤场景下，也可以用 BloomFilter 做快速拦截。

2. 防击穿。
对热点数据引入 Caffeine 本地缓存和 HotKey 热点识别，不让单个热点 key 失效后所有请求都直冲数据库；必要时可以配合互斥更新或短暂降级。

3. 防雪崩。
缓存过期时间不要完全一致，同时在系统层做 Sentinel 限流和降级，一旦缓存层出问题，至少能控制请求不要把数据库拖死。

所以我理解这三类问题的关键，不是背定义，而是知道它们分别对应“无效请求”“单热点请求”“全局失效请求”。

---

### 22. HotKey 是为了解决什么问题？和普通 Redis 热点缓存相比，它的价值是什么？

HotKey 解决的是“热点数据访问高度集中，导致单个 key 被频繁访问，从而把 Redis、数据库甚至整个链路打热”的问题。

普通 Redis 热点缓存的本质，还是把热点流量集中到 Redis 这一层。如果访问量再往上走，Redis 自己也会变成瓶颈。尤其是某些爆款题库、爆款题目详情页，所有流量都打到同一个 key，上层虽然不查数据库了，但 Redis 压力仍然可能很大。

HotKey 的价值在于，它不是简单地把热点放进 Redis，而是通过热点识别，把真正的超高频 key 提升到本地缓存层。这样很多请求在应用内存层就结束了，根本不会再经过 Redis。

所以相比“普通 Redis 缓存”，HotKey 的核心增量价值是：

1. 能动态识别热点，而不是完全靠人工预估；
2. 能把热点请求从远程缓存继续下沉到本地缓存；
3. 在热点波动场景下，更适合应对瞬时流量。

如果一句话概括，我会说：Redis 是缓存热点，HotKey 是治理热点。

---

### 23. 如果出现热 key，把请求都打到单个 Redis key 上，会有什么问题？你怎么优化？

如果所有请求都集中打到单个 Redis key，上游虽然不查数据库了，但新的瓶颈会转移到 Redis 本身。表现上可能是这个 key 所在分片压力升高、RT 波动、网络 IO 放大，严重时还会影响同实例上的其他业务。

这个问题的本质不是“有没有缓存”，而是“缓存层本身是否成了单点热点”。

我的优化思路通常分三层。

第一层，把真正超高频的 key 下沉到本地缓存，用 Caffeine 或 HotKey 让大部分请求在应用内存层命中。

第二层，对访问链路做限流和保护。如果某个热点请求突然暴涨，可以用 Sentinel 做限流，优先保护系统整体稳定。

第三层，必要时做数据拆分或降级。比如对某些热点榜单不追求强实时，可以用预计算和短 TTL 缓存，降低实时回源压力。

所以热 key 问题本质上是“热点集中度”问题，解决办法也不是单一加缓存，而是缓存下沉、流量治理和降级兜底一起做。

---

## 四、搜索与 ES 基础追问

### 24. ES 的倒排索引原理你怎么理解？

我一般会把倒排索引讲成一句很容易理解的话：普通关系型查询更像是“我拿着一条数据去找它有哪些字段”，而倒排索引更像是“我拿着一个词，直接找到它出现在哪些文档里”。

比如有三道题目：

1. Redis 热 key
2. MySQL 索引优化
3. Elasticsearch 倒排索引

如果用倒排索引，系统会先把“Redis”“热”“key”“MySQL”“索引”“Elasticsearch”等词拆出来，再建立“词 -> 文档集合”的映射。这样用户搜索“索引”时，系统不需要把每条题目全文扫描一遍，而是直接从索引里拿到包含“索引”的文档集合。

所以倒排索引快的根本原因，不是因为 ES 魔法更强，而是因为它把搜索成本前置到了建索引阶段。搜索时做的是索引查找，而不是全文遍历。

这也是为什么 ES 很适合全文检索，但写入成本通常会比单纯写 MySQL 更高，因为它在写入时就要维护索引结构。

---

### 25. 题目搜索时，你们搜的是哪些字段？分词策略是什么？

我这个项目里，题目搜索主要围绕几个最有业务价值的字段来做：

1. 题目标题；
2. 题目内容；
3. 标签；
4. 难度这类可精确过滤字段。

其中标题和内容是典型的全文检索字段，适合走分词；标签通常既可以做精确过滤，也可以做一定程度的检索扩展；难度更适合做 term 过滤，而不是全文搜索。

分词策略上，我会把标题和内容这类中文文本字段接 IK 分词器，让它们在写入和查询阶段都按中文词项进行切分。这样用户搜“Redis 热 key”时，不是用整句硬匹配，而是把关键词拆开后再做倒排检索。

如果面试官继续追问，我会补一句：不是所有字段都应该分词。像题目 ID、难度、状态、是否删除这类字段就更适合走精确匹配，不应该按全文检索来设计。

---

### 26. 为什么标签、题目内容这类字段适合放 ES？

因为这类字段有两个典型特点：第一是文本性强，第二是用户会拿它们做搜索。

题目内容通常篇幅比较长，信息量大，天然适合全文检索。标签虽然看起来比内容更结构化，但本质上它承载的是主题和知识点语义，用户往往也会拿它做搜索、筛选或组合查询。

这类字段放到 ES 里以后，可以利用倒排索引、分词器和相关性排序能力，把“搜题”这件事做得比数据库模糊匹配更像真正的搜索引擎。

相反，那些偏事务、偏结构化、偏更新频繁但检索价值不高的字段，就不一定适合放进 ES 核心检索链路里，否则只会增加同步成本。

---

### 27. ES 查询和 MySQL 查询的适用场景分别是什么？

我会把它们分得很清楚。

MySQL 更适合结构化查询和强事务场景。比如管理员新增题目、修改题库、按主键查询详情、按状态筛选后台数据，这些都更适合 MySQL，因为它的数据一致性、事务能力和关系建模能力更强。

ES 更适合全文检索和复杂搜索场景。比如用户按关键词搜题、按标签和关键字组合搜索、做相关性排序，这些都是 ES 的优势场景。

所以在我的项目里，MySQL 是“事实数据源”，ES 是“检索加速层”。我不会让 ES 取代 MySQL 去承担主存储职责，也不会让 MySQL 去硬扛全文检索职责。

---

### 28. 如果搜索结果不准，你会从 mapping、分词器、查询 DSL 哪几层排查？

如果搜索结果不准，我一般按三层排查。

第一层看 mapping。
我要先确认字段类型是不是设计对了，比如该分词的字段是不是 `text`，该精确过滤的字段是不是 `keyword` 或者适合 term 查询的类型。如果 mapping 从一开始就错了，那后面的查询再怎么调也很难对。

第二层看分词器。
尤其是中文搜索，很多“不准”其实不是查询逻辑错，而是分词效果不好。比如某个技术词被切碎了，或者查询词和索引词切分方式不一致，都会导致召回异常。所以我会用分词测试工具先看实际分词结果。

第三层看查询 DSL。
比如是 `match` 还是 `term`，是 `must`、`should` 还是 `filter`，有没有把精确过滤和全文检索混在一起，排序逻辑是不是把相关性压掉了。这些都会直接影响最终结果。

所以我的排查顺序是：先看数据结构，再看词项切分，最后看查询表达式。这样效率最高，也最符合问题定位逻辑。

---

### 29. ES 为什么查询快，但不适合完全替代 MySQL 做主存储？

因为 ES 和 MySQL 的设计目标就不一样。

ES 快，是因为它针对全文检索做了大量优化，比如倒排索引、分词、相关性排序等，所以在搜索场景下性能非常强。

但它不适合完全替代 MySQL 做主存储，主要有几个原因。

第一，事务能力弱。MySQL 天生更适合承接强一致、强事务场景。

第二，关系建模能力弱。像题库和题目的多对多关系、后台管理数据的结构化操作，MySQL 更自然。

第三，写入和索引维护成本高。ES 的每次写入都不仅仅是“存数据”，还要维护索引结构。

第四，运维和一致性复杂度更高。把 ES 当检索副本比较合理，把它当主存储会把很多复杂问题提前暴露出来。

所以在我这个项目里，ES 的定位非常清晰：它是搜索引擎，不是主数据库。

---

## 五、安全与稳定性高频追问

### 30. 你项目里为什么要引入 Sentinel？它主要解决什么问题？

我引入 Sentinel，核心不是为了“展示我会一个中间件”，而是因为这个项目本身存在明显的高频查询接口和热点访问链路，如果没有流量治理能力，系统很容易在突发流量或者依赖异常时失控。

Sentinel 主要帮我解决三类问题。

第一，限流。比如题目列表、题库列表这种高频接口，在流量突然升高时，需要控制进入系统的请求速度。

第二，熔断。当接口异常率过高或者慢调用过多时，系统不能继续放任请求堆积，而是要主动切断故障链路。

第三，降级。也就是在依赖不稳定或系统压力大时，优先保证核心服务还能返回一个可接受的结果，比如本地缓存兜底。

所以 Sentinel 的本质价值，是把“系统被动承压”变成“系统主动治理流量和故障”。

---

### 31. 你用了哪些限流规则？QPS 限流、热点参数限流、熔断降级分别是什么？

在这个项目里，我主要会围绕核心查询接口配置几类规则。

第一类是普通 QPS 限流。
它控制的是某个接口单位时间内允许通过的请求数。适合题库列表、题目列表这种整体流量大的接口，目的是防止瞬时洪峰把服务压垮。

第二类是热点参数限流。
这类规则关注的不是“接口整体有多热”，而是“某个特定参数是不是特别热”。比如某个热门题库 ID、某个热门搜索条件，如果所有请求都集中在它身上，就会形成局部热点。热点参数限流就是针对这种“局部过热”的情况。

第三类是熔断降级。
它不是简单限制流量，而是在异常率、慢调用比例或响应时间达到阈值时，主动把接口切到保护状态，避免故障继续放大。

如果要一句话区分，我会说：

1. QPS 限流解决“总量过大”；
2. 热点参数限流解决“局部过热”；
3. 熔断降级解决“接口已经不稳定了，不能再继续硬扛”。

---

### 32. 你项目里是怎么做“被限流后兜底”的？为什么用本地缓存兜底？

我在这个项目里的思路是：限流不是目的，系统保护之后仍尽量让用户拿到“还能用的数据”。

具体来说，对于题库列表、热门题目等高频读接口，我会在正常链路下维护一份本地缓存。当 Sentinel 触发限流或降级时，优先直接返回本地缓存中的最近结果，而不是简单地返回报错。

为什么我更倾向于用本地缓存兜底，而不是继续去查 Redis？

因为在限流和降级触发的场景下，我优先考虑的是“减少外部依赖”。本地缓存就在应用内存中，没有网络开销，也不会把压力再转移给 Redis。尤其是在热点流量场景下，如果 Redis 本身也承压，那继续访问 Redis 并不是真正的兜底。

所以用本地缓存兜底的核心逻辑是：在非常态场景下，优先用最轻量、最稳定的方式保住核心功能。

---

### 33. 限流和熔断有什么区别？

限流和熔断的触发背景不一样。

限流更偏“流量太大了，我要控制进入量”，它关注的是外部请求强度；

熔断更偏“接口已经不稳定了，我不能再继续放请求进去”，它关注的是内部服务健康状态。

也就是说，限流是防御流量洪峰，熔断是防止故障扩散。

如果要举个项目里的例子：

1. 某个题目列表接口平时很稳定，但突然流量暴增，这时候应该优先限流。
2. 如果某个接口已经出现明显慢调用或者错误率升高，这时候即使流量不算特别大，也应该熔断保护。

所以两者经常一起用，但解决的问题并不是同一个层面。

---

### 34. 为什么要做动态 IP 黑白名单？为什么配合 Nacos？

做动态 IP 黑白名单，核心原因是安全规则和业务代码不应该强耦合。

像恶意 IP、爬虫 IP、异常访问源这种东西，变化往往比较快。如果每次更新名单都要改代码、发版、重启，那响应速度太慢，也不适合真实线上场景。

所以我会把这类规则从代码里抽出来，放进配置中心管理。配合 Nacos 的好处主要有两个。

第一，可以动态下发配置，不需要重新部署服务。

第二，多实例环境下规则可以统一分发，保证各个节点看到的名单是一致的。

所以 Nacos 在这里的价值，不只是“有个地方能配配置”，而是让安全规则具备动态治理能力。

---

### 35. 你黑名单为什么用 BloomFilter，而不是 Set<String>？

如果只是少量 IP，用 Set 当然最直观，也没有误判问题。但一旦黑名单量变大，或者我希望它在高频请求判断中尽量节省内存和提升吞吐，BloomFilter 就更有优势。

我用 BloomFilter 的核心原因是：

1. 内存占用小；
2. 查询速度快；
3. 很适合做“快速判断是否疑似命中”的第一层过滤。

当然，BloomFilter 的缺点我也会主动说，就是它有误判率，而且删除不方便。所以它不是绝对精确的数据结构，更适合作为高频防护场景中的快速筛子。

如果面试官继续追问，我会补一句：在实际系统里，BloomFilter 最适合放在前面挡掉大部分明显无效或高风险请求，精确裁决仍然可以交给后续更严格的规则。

---

### 36. Sa-Token 在你项目里具体解决了什么问题？和 Spring Security 比你为什么选它？

Sa-Token 在我的项目里主要解决两件事：

第一，登录态管理。也就是用户登录、校验是否已登录、退出登录、获取当前登录用户这类典型认证问题。

第二，权限控制。比如管理员接口和普通用户接口要区分访问权限，以及一些会话控制能力，比如同端登录互斥。

我选 Sa-Token 而不是 Spring Security，主要是考虑到这个项目的复杂度和开发效率。

Spring Security 很强，但配置和概念体系也更重，适合更复杂、更标准化的大型安全体系；Sa-Token 更轻量，上手快，接口也更直接。对于刷题平台这种中小型业务系统，它能很好满足认证、授权和会话控制需求，而且开发成本更低。

如果一句话总结，我会说：不是 Spring Security 不好，而是 Sa-Token 在这个项目里是“更合适的工程选择”。

---

### 37. 什么叫“同端登录互斥”？你是怎么实现的？

同端登录互斥，指的是同一个账号在同一个设备类型上，只允许保留一个有效登录会话。比如用户在 Web 端重新登录后，旧的 Web 端会话会被挤掉。

这个设计的价值主要有两个。

第一，减少账号共享或异常并发登录带来的风险。

第二，提升会话管理的可控性，比如后续要做风险下线、踢出登录时更容易落地。

实现上，我是借助 Sa-Token 的登录能力和设备维度标识来做的。简单说，就是登录时带上设备标识，让框架按设备维度维护登录态；当同一账号同一设备再次登录时，旧会话会被替换或踢出。

所以这个功能本质上是“基于设备维度管理会话唯一性”，而不是简单地禁止多端登录。

---

### 38. 你们的反爬虫策略是怎么设计的？为什么要分级而不是一刀切封禁？

我不太赞成一开始就一刀切封禁，因为真实业务里，用户高频访问不一定都是恶意，也可能是正常刷题行为、接口重试、网络抖动等因素造成的。

所以我的反爬策略更偏“分级治理”。

第一层是访问频控。先对用户或 IP 的访问频率做统计。

第二层是告警。如果达到风险阈值，先记录日志或者触发告警，而不是立刻封。

第三层是限制。对于明显异常的访问，先做限流或短时封禁。

第四层是升级处理。对于持续高风险行为，再做更强的拦截，比如加入黑名单、强制下线、封号等。

之所以这样设计，是因为反爬的目标不是“尽可能多封”，而是“尽可能少误伤地保护系统和内容”。

---

### 39. 如果误伤正常用户怎么办？你会怎么优化反爬策略？

如果误伤正常用户，我会先承认：这类系统没有绝对零误伤，关键是要让误伤概率低、恢复成本低、策略可调。

我的优化思路一般有几条。

第一，不只看单一指标。不要只因为一次高频访问就直接封禁，而要结合时间窗口、访问路径、账号行为、是否持续触发等多个条件综合判断。

第二，采用分级策略。先告警、再限流、再短时限制、最后才是永久封禁，逐级升级。

第三，保留人工恢复和白名单通道。对于被误伤的正常用户，可以快速解除限制。

第四，把规则做成动态可调。比如把阈值放到配置中心，通过 Nacos 动态调整，而不是写死在代码里。

所以我的核心原则是：反爬不是只追求拦截率，还要兼顾用户体验和运营可控性。

## 六、数据库 / 框架基础追问

### 40. 为什么选 MyBatis-Plus？它帮你解决了什么问题？

我选 MyBatis-Plus，主要是因为它在保留 MyBatis 灵活性的同时，大幅减少了样板代码。

对于这个项目来说，有很多后台管理和基础数据操作，如果全靠原生 MyBatis 去写 CRUD，会有大量重复 Mapper 和 XML 代码。MyBatis-Plus 提供了通用 Service、条件构造器、分页插件等能力，让我可以把精力更多放在业务查询和性能设计上，而不是机械写增删改查。

它帮我解决的最直接问题有三个：

1. 基础 CRUD 开发效率；
2. 条件查询拼接的可读性；
3. 分页和逻辑删除这些常见功能的快速接入。

所以我会把它定义成“提高基础开发效率的工程增强层”，而不是替代 SQL 能力本身。

---

### 41. 分页插件底层是怎么实现的？

MyBatis-Plus 的分页插件本质上还是拦截 SQL，然后对原始查询做分页改写。

一般来说，它会在执行查询前拦截当前语句，根据分页参数拼接数据库方言对应的分页 SQL，比如 MySQL 下常见的 `limit offset, size`。同时，它通常还会额外生成一条 `count` 语句，用来统计总记录数。

所以它底层并不神秘，本质上就是“SQL 拦截 + 分页 SQL 改写 + 总数统计”。

不过我也会补一句：分页插件虽然方便，但如果原始 SQL 本身很复杂、关联很多表、count 成本很高，分页插件也不能从根本上替你解决性能问题，最终还是要回到 SQL 设计和索引设计上。

---

### 42. 逻辑删除和物理删除的区别是什么？什么时候用哪种？

逻辑删除是指数据不真正删除，而是打一个删除标记，比如 `isDelete = 1`；物理删除则是直接把数据从数据库中删掉。

逻辑删除适合这几类场景：

1. 数据有恢复需求；
2. 需要保留操作痕迹；
3. 业务上仍希望留存历史记录。

物理删除更适合：

1. 明确无价值的临时数据；
2. 明确要求释放存储空间的数据；
3. 法务或数据治理要求必须彻底清除的数据。

在我的项目里，题目、题库、用户这类核心业务数据更适合逻辑删除，因为后台管理和问题排查中都可能需要恢复或追溯。

---

### 43. 事务的传播行为你了解哪些？你项目里的批处理为什么要关注事务边界？

常见传播行为里，我面试时一般重点讲 `REQUIRED`、`REQUIRES_NEW`、`SUPPORTS` 这几个就够了。

`REQUIRED` 是默认值，表示当前有事务就加入，没有就新建；
`REQUIRES_NEW` 是无论外面有没有事务，都新开一个独立事务；
`SUPPORTS` 是有事务就加入，没有也可以非事务运行。

我项目里的批处理为什么要特别关注事务边界？因为批量操作一旦全塞进一个大事务里，事务时间会很长，失败回滚范围也会很大，数据库锁持有时间也会更长。

所以我会把一个超大批量任务拆成多个小批次，每个批次各自用事务保证内部一致性，而不是追求全局一个超大事务。这样更符合高并发和大批量场景下的稳定性要求。

---

### 44. 为什么连接池选 Druid？它相比 HikariCP 的优势和不足是什么？

我选 Druid，主要是因为它在监控和 SQL 分析方面的能力比较强，尤其适合教学项目或者需要观察 SQL 行为的场景。

它的优势主要在于：

1. 内置监控能力强；
2. 慢 SQL 统计和可视化支持好；
3. 对连接池状态、SQL 执行情况更容易做运维观察。

它的不足也很明确：

1. 纯性能角度通常不如 HikariCP 轻量；
2. 配置项更多，复杂度更高；
3. 在一些极致性能场景里，HikariCP 往往更常见。

所以如果是一个强调监控可观测性的项目，Druid 很合适；如果是极致轻量和高性能优先，也可以考虑 HikariCP。我在这个项目里更看重它的监控能力。

---

### 45. 慢 SQL 你怎么排查？项目里如何监控数据库性能？

我一般会从三个层面排查慢 SQL。

第一层看现象。比如接口 RT 变慢、数据库 CPU 飙高、连接池活跃连接增多。

第二层看 SQL 语句本身。通过 Druid 的慢 SQL 统计或者日志，找到具体慢在哪条 SQL。

第三层看执行计划。重点关注有没有走索引、是不是出现了全表扫描、回表过多、排序或分页代价高等问题。

项目里监控数据库性能，我会关注这些指标：

1. 慢 SQL 数量；
2. 平均响应时间和 TP99；
3. 活跃连接数、等待连接数；
4. 读写比例和热点 SQL；
5. 索引命中情况。

所以我理解慢 SQL 排查，不是只看“语句慢”，而是要把接口、连接池和 SQL 执行计划串起来看。

---

### 46. 你做过哪些索引设计？搜索、排序、分页时索引怎么考虑？

在这个项目里，MySQL 侧的索引设计更多是围绕结构化查询来做，而全文检索交给 ES。

比如：

1. 主键和关联字段一定要有索引；
2. 题库题目关联表里的题库 ID、题目 ID 这种高频关联字段要重点加索引；
3. 管理后台常用过滤字段、排序字段要结合实际查询条件考虑联合索引。

对于排序和分页，我会特别注意两点。

第一，排序字段如果没有合适索引，代价会很大；

第二，深分页本身就贵，无论 MySQL 还是 ES 都一样，所以要控制分页深度，或者使用更适合的分页方式。

而搜索这块我不会强行在 MySQL 上追求复杂全文能力，而是明确交给 ES 去处理。这个边界划清后，索引设计会更合理。

## 七、前端高频追问

### 47. 为什么 Next.js 适合这个项目？SSR 对 SEO、首屏性能有什么帮助？

Next.js 适合这个项目，最核心的原因是它非常适合内容型、列表型、详情型页面较多的场景。

像题库页、题目详情页、搜索结果页，本质上都属于“用户进入页面时就希望尽快看到内容”的页面。SSR 可以让服务端先把页面内容渲染出来，再返回给浏览器，这样用户首屏看到内容更快，也更利于搜索引擎抓取。

SEO 方面，SSR 的优势在于首屏 HTML 里就已经包含主要内容，而不是等浏览器执行大量 JS 后再生成页面，这对内容平台更友好。

首屏性能方面，SSR 能降低用户感知到的白屏时间，特别是在网络条件一般或者设备性能一般时，体验差异会更明显。

所以我选 Next.js，不是因为它“更潮”，而是因为它和刷题平台这种内容型业务在渲染模型上更匹配。

### 48. 你项目里哪些页面适合 SSR，哪些更适合 CSR？

我一般按“是否重 SEO、是否重首屏、是否强交互”来区分。

适合 SSR 的页面有：

1. 首页；
2. 题库列表页；
3. 题目详情页；
4. 搜索结果页。

因为这些页面更看重首屏内容展示，也更可能被搜索引擎抓取。

更适合 CSR 的页面有：

1. 用户中心；
2. 后台管理页；
3. 高交互表单页；
4. AI 模拟面试聊天页。

因为这些页面通常依赖登录态、交互频繁、状态变化多，纯客户端渲染反而更自然。

所以不是所有页面都硬上 SSR，而是按业务目标做混合渲染。

### 49. Redux 在这个项目里主要存了什么状态？为什么不全靠组件状态？

Redux 在这个项目里主要存的是“全局共享、跨页面有价值”的状态，比如当前登录用户信息、权限相关状态、可能还有一些全局菜单或布局级状态。

为什么不全靠组件状态？因为组件状态适合局部、短生命周期的数据，但像登录用户这种信息，很多页面和组件都会依赖。如果每个页面都自己重新请求、自己维护，不但代码重复，也容易出现状态不一致。

所以 Redux 更适合承接“全局公共状态”，组件本地状态更适合承接“当前页面临时交互状态”。这是一个职责边界问题，不是说 Redux 越多越好。

---

### 50. 你怎么处理登录态初始化？首屏时如果用户未登录，页面怎么保证不报错？

我的处理思路是：应用初始化时先请求当前登录用户接口，把登录态放进全局状态；如果请求失败或者返回未登录，就写入一个默认游客态，而不是让页面直接拿空对象去渲染。

这样做的好处是，页面层可以统一面向“有一个稳定的用户状态对象”来开发，不需要到处写空判断。

同时，在 Axios 拦截器或者统一请求层，我还可以对未登录错误码做统一处理，比如在需要登录的页面跳转到登录页，在不需要登录的页面保持游客态继续浏览。

所以本质上我做的是“先把登录态抽象成稳定状态模型，再让页面按这个模型工作”，这样首屏未登录也不会直接报错。

---

### 51. 前端接口代码为什么用 OpenAPI 生成？这样做有什么收益和代价？

我用 OpenAPI 生成前端接口代码，主要是为了减少手写请求层的重复劳动，并且让前后端接口定义更一致。

收益很直接：

1. 接口方法和类型定义自动生成，减少重复写样板代码；
2. 参数和返回结构更统一，类型提示更好；
3. 后端接口变更后，前端可以更快同步。

代价也有：

1. 生成代码会引入额外工程流程；
2. 如果后端文档维护不规范，生成结果质量也会受影响；
3. 有时候前端还需要在生成代码基础上再做一层封装，不能完全拿来即用。

所以我的看法是，OpenAPI 更适合“接口较多、协作频繁、追求一致性”的项目，在这个项目里是值得的。

---

### 52. 你是怎么做权限控制和菜单控制的？

我的思路是把“权限判断”和“菜单展示”都建立在统一的用户角色模型上。

后端负责真正的权限校验，比如管理员接口必须校验管理员角色，这个是安全边界，必须在服务端兜底。

前端负责的是展示层权限控制，比如根据当前登录用户角色动态生成可见菜单、控制某些按钮是否显示、未授权页面是否跳转或提示无权限。

所以本质上是“双层控制”：

1. 后端做真正的安全校验；
2. 前端做用户体验层的权限感知。

这样既保证安全，也保证交互自然。

---

### 53. Axios 拦截器在你项目里做了什么事情？

Axios 拦截器主要承担统一请求治理的职责。

请求拦截器这边，可以统一加基础配置，比如 baseURL、cookie、超时等；

响应拦截器这边，主要做两件事：

1. 统一解析后端返回结构，比如 `code`、`message`、`data`；
2. 统一处理未登录、业务异常和通用错误。

比如后端返回未登录错误码时，我可以在拦截器里统一判断：如果当前页面需要登录，就跳去登录页；如果是公共页面，就按游客态处理。

这样做的价值在于：把通用请求逻辑从业务页面里抽出来，让页面层更关注“拿到什么数据后怎么展示”，而不是到处写重复异常处理。

## 八、AI 模块如果写到简历，几乎一定会问

### 54. AI 模拟面试这个功能的业务流程是什么？

这个功能我会把它讲成一条完整业务链路，而不是“调用了一次大模型接口”。

整体流程大概是：

1. 用户选择题库、难度或者面试方向；
2. 系统根据这些条件从题库中筛选候选题目；
3. 大模型通过 Function Calling 调用后端能力，完成选题、出题和追问；
4. 用户作答后，系统结合题目标准答案、知识点和用户回答，让模型做针对性点评；
5. 最终生成结构化面试报告，包括表现总结、薄弱点、建议和后续复习方向。

这里的关键点是，模型不是完全自由发挥，而是被题库、难度和标准答案这些业务数据约束着工作，所以结果会更可控。

---

### 55. 大模型接入时，你怎么设计 prompt？

我设计 prompt 时最关注的不是“写得多华丽”，而是让模型角色、目标、边界和输出格式都足够明确。

一般会明确这几件事：

1. 角色：你是面试官，不是聊天助手；
2. 目标：围绕指定题库和难度做面试提问与点评；
3. 约束：题目来源要受控，点评要结合标准答案，不能偏离技术面试场景；
4. 输出：尽量按结构化字段输出，比如问题、评价、建议、得分维度等。

如果结合 Function Calling，那 prompt 的作用更偏“定义模型应该在什么情况下调用什么能力、调用后怎么组织结果”，而不是单纯让它自由生成一大段文本。

所以我的 prompt 设计重点是：明确角色、限制范围、固定输出，而不是追求文学性。

---

### 56. 如果模型返回结果格式不稳定，你怎么兜底？

这是 AI 业务里非常现实的问题，所以我不会假设模型每次都 perfectly follow instruction。

我的兜底思路一般分三层。

第一层，优先用 Function Calling 或结构化 schema 约束输出，让模型尽量按预期格式返回。

第二层，服务端做解析和校验。比如字段缺失、结构不完整时，先做修正或回退，而不是直接把原始结果透给前端。

第三层，准备降级方案。如果结构化结果拿不到，就退化成普通文本总结，至少让用户还能拿到一个可读结果。

所以在 AI 场景里，我不会把模型输出当成绝对可信的后端返回值，而是把它当作“需要经过系统再加工的一种半结构化结果”。

---

### 57. AI 生成题目为什么要做“待审核”而不是直接入库？

因为 AI 生成内容虽然很高效，但不代表一定准确、严谨、符合平台标准。

如果直接入库，可能会带来几个风险：

1. 技术表述不准确；
2. 难度不匹配；
3. 题目和平台风格不一致；
4. 甚至可能出现重复题、低质量题或者幻觉内容。

所以更合理的做法是把 AI 生成题目标记为“待审核”，先让管理员或运营确认质量后再正式进入题库。

这本质上是在 AI 提效和内容质量之间做平衡。AI 负责提草稿和初稿，人来做最终质量把关。

---

### 58. 如果 AI 接口超时或失败，你怎么处理重试、降级和用户体验？

AI 接口超时或失败时，我会把它当成一个不稳定依赖来处理，而不是当成核心强依赖。

第一，超时控制一定要有，不能无限等。否则用户体验会非常差，也会拖住线程资源。

第二，重试要有边界。像瞬时网络抖动可以做有限次重试，但不能无脑重试，否则只会把依赖雪上加霜。

第三，要有清晰降级。比如告诉用户“当前 AI 服务繁忙，请稍后重试”，或者保留已有题库练习能力，不让整个面试模块彻底不可用。

第四，前端交互要友好。比如给出加载状态、失败提示、可重试按钮，而不是一直转圈。

所以我的原则是：AI 很重要，但它在系统里仍然是一个需要被保护、被限时、被降级的外部能力。

## 九、结尾建议

### 59. 这些题在面试时怎么答效果最好？

我建议你始终按这个结构答：

1. 先说业务场景；
2. 再说原始问题；
3. 再说你的设计方案；
4. 最后补一句方案权衡。

比如不要只说“我用了 ES”，而要说“因为 MySQL like 在题目搜索场景下性能和中文检索效果都不够，所以我引入 ES，并配合 IK 分词和定时同步来解决性能和一致性问题”。

这样面试官会更容易判断你是真的做过设计，而不是只会背名词。

### 60. 哪几道题最值得你优先背熟？

如果时间有限，我建议你最先背熟这几类：

1. 项目 1 分钟介绍；
2. 为什么用 ES 替代 MySQL like；
3. MySQL 和 ES 的最终一致性方案；
4. Redis + Caffeine + HotKey 的缓存设计；
5. Sentinel 的限流、熔断和本地缓存兜底；
6. Redis BitMap 为什么适合刷题记录；
7. AI 模拟面试的业务闭环和 Function Calling 设计。

这几题一旦答顺了，面试官对你这个项目的整体印象通常就会比较强。
